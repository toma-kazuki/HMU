## Background — Two parallel data representations causing naming drift and broken UI

### Current state

The codebase has two overlapping layers for wearable data:

**Layer A — `WearableSnapshot` (legacy aggregate)**
- A flat struct (`models.py:47`) holding 12 fields: `heart_rate_bpm`, `spo2_pct`, `respiratory_rate_bpm`, `skin_temp_c`, `systolic_mmhg`, `diastolic_mmhg`, plus the six synthesized scores.
- Generated by `mock_data.build_wearable()`.
- Field name: `respiratory_rate_bpm`.

**Layer B — `WearableDevices` (device-native structs)**
- Six typed structs (`BioMonitorData`, `OuraRingData`, etc.) holding all 35 sensor parameters listed in `parameter_symptom.csv`.
- Generated by `mock_data.build_devices()`.
- The same respiratory measurement on `BioMonitorData` is named `breathing_rate_bpm`.

**Layer C — `VitalSample` / `vitals_series` (legacy timeseries)**
- A flat 3-field timeseries (`heart_rate_bpm`, `spo2_pct`, `respiratory_rate_bpm`).
- Generated by `mock_data.vitals_timeseries()`.
- Referenced in `ALERT_TREND_MAP` for the SpO₂, Heart rate, and Respiratory rate chart entries.

`mock_data.build_devices()` copies values from Layer A into Layer B at lines 261–266 and 289, creating a secondary derivation dependency. `WearableSnapshot` and `VitalSample` are architectural dead weight: they predate the per-device model design and serve no function that the device structs do not already cover.

### Root cause / Gap description

The name mismatch between Layer A/C (`respiratory_rate_bpm`) and Layer B (`breathing_rate_bpm`) caused `AlertItem.parameter = "Breathing rate"` (set in `alerts.py`) to fail lookup in `ALERT_TREND_MAP['Respiratory rate']` (which draws from `vitals_series.respiratory_rate_bpm`).

```
alerts.py _evaluate_breathing_rate():
    → _make_alert(parameter="Breathing rate", ...)

app.js ALERT_TREND_MAP:
    'Respiratory rate': { getSeries: d => d.vitals_series.map(s => s.respiratory_rate_bpm) }

ALERT_TREND_MAP["Breathing rate"]  →  undefined  →  no chart rendered
```

The fix is not a simple alias. The root cause is the two-layer architecture. The correct fix is to eliminate `WearableSnapshot` and `VitalSample` entirely and make `WearableDevices` (Layer B) the sole source of truth.

### Symptoms or evidence

- Bradypnoea alert panel shows no trend chart (screenshot in conversation).
- Tachypnoea alert panel also has no trend chart for the same reason.
- `ALERT_TREND_MAP['Respiratory rate']` entry would also fail for any alert that uses a different `parameter` string.
- `mock_data.build_devices()` lines 261–266, 289 silently copy `WearableSnapshot` fields into device structs — device values are not independently generated, defeating the purpose of the per-device model.

## Task 2 — Rewrite `mock_data.py`: device-native generation, remove `vitals_timeseries`

**Status:** 🔲 Not started  
**Prerequisites:** Task 1

### Objective

`build_devices()` currently copies values from `WearableSnapshot` (Layer A) into device structs. After Task 1 removes `WearableSnapshot`, `build_devices()` must generate all values independently. Score computation moves to a new `build_scores()` function. `vitals_timeseries()` is deleted.

### Design references

- `7_parameter_logic.md §4` — score computation logic (which inputs, which formula)
- `mock_data.py` lines 142–353 — existing generation logic to preserve scenario/seed behaviour

### Files to modify

- `backend/mock_data.py`

### Changes required

**Step A — Rename and rewrite `build_wearable()` → `build_devices()`**

The existing `build_devices(crew_member_id, mission_day, wearable, *, scenario)` takes `wearable: WearableSnapshot` as input. The new signature is:

```python
def build_devices(
    crew_member_id: str,
    mission_day: int,
    *,
    scenario: str = "nominal",
) -> WearableDevices:
```

All values now generated independently. Fields previously copied from `wearable.*` must be generated inline:

| Old copy | New generation |
| :--- | :--- |
| `wearable.heart_rate_bpm` | generate `hr` inline (same formula as old `build_wearable`) |
| `wearable.spo2_pct` | generate `spo2` inline |
| `wearable.respiratory_rate_bpm` | generate `rr` inline; field is `breathing_rate_bpm` on `BioMonitorData` |
| `wearable.systolic_mmhg` | generate `sys_bp` inline |
| `wearable.diastolic_mmhg` | generate `dia_bp` inline |
| `wearable.skin_temp_c` | generate `skin` inline |

Alert demo overrides (`_alert_demo and crew_member_id == _ALERT_DEMO_CREW`) must remain:
```python
if _alert_demo and crew_member_id == _ALERT_DEMO_CREW:
    spo2   = 88.5
    hr     = 128.0
    sys_bp = 168.0
    rr     = 7.0    # → bio_monitor.breathing_rate_bpm
```

Oura ring `respiratory_rate_bpm` is independently generated (slight offset from `rr`); keep as-is.

**Step B — Add `build_scores()`**

```python
def build_scores(
    crew_member_id: str,
    mission_day: int,
    devices: WearableDevices,
    *,
    scenario: str = "nominal",
) -> ScoreSnapshot:
```

Move score computation from old `build_wearable()` lines 163–179 here. Use `devices` fields as inputs where the design specifies (e.g. HRV from `devices.oura_ring.hrv_ms`). Seed RNG identically to preserve determinism.

**Step C — Delete `vitals_timeseries()` (lines 391–414) and `VitalSample` import**

**Step D — Update `build_integrity()` signature**

`build_integrity()` currently takes `wearable: WearableSnapshot` and reads `wearable.wearable_link_quality_pct` and `wearable.last_sync_ago_sec`. Move these fields to `BioMonitorData` or derive from `DeviceStatus`. Simplest: read from `devices.bio_monitor.status`:

```python
def build_integrity(devices: WearableDevices, *, force_degraded: bool | None = None) -> SensorIntegrity:
    degraded = _demo_degraded if force_degraded is None else force_degraded
    if degraded:
        return SensorIntegrity(heart_rate="stale", spo2="ok", environmental="stale")
    bio_signal = devices.bio_monitor.status.signal
    if bio_signal == "lost":
        return SensorIntegrity(heart_rate="lost", spo2="lost", environmental="ok")
    if bio_signal == "stale":
        return SensorIntegrity(heart_rate="stale", spo2="stale", environmental="ok")
    return SensorIntegrity(heart_rate="ok", spo2="ok", environmental="ok")
```

Remove `wearable_link_quality_pct` and `last_sync_ago_sec` from `BioMonitorData` (they were `WearableSnapshot` fields, not real sensor parameters). Note: `DeviceStatus.signal` and `DeviceStatus.last_sync_ago_sec` already exist on every device — use those.

## Task 3 — Update `main.py`: remove legacy fields from payload construction

**Status:** 🔲 Not started  
**Prerequisites:** Task 2

### Objective

`main.py` currently calls `build_wearable()`, passes the result to `build_devices()` and `build_integrity()`, then packs `wearable=wearable` and `vitals_series=series` into `DashboardPayload`. After Tasks 1–2, the call chain becomes simpler.

### Files to modify

- `backend/main.py`

### Changes required

**Step A — Update detail endpoint call chain**

Old:
```python
wearable = mock_data.build_wearable(crew_member_id, mission_day, scenario=scenario)
devices  = mock_data.build_devices(crew_member_id, mission_day, wearable, scenario=scenario)
integrity = mock_data.build_integrity(wearable, ...)
series   = mock_data.vitals_timeseries(wearable.heart_rate_bpm, wearable.spo2_pct, wearable.respiratory_rate_bpm)
```

New:
```python
devices   = mock_data.build_devices(crew_member_id, mission_day, scenario=scenario)
scores    = mock_data.build_scores(crew_member_id, mission_day, devices, scenario=scenario)
integrity = mock_data.build_integrity(devices)
```

**Step B — Update `DashboardPayload` construction**

Remove `wearable=wearable` and `vitals_series=series`; add `scores=scores`.

**Step C — Update overview endpoint**

The overview endpoint reads score fields from `wearable.*` to populate `CrewColumnSummary`. Update to read from `scores.*`:
```python
health_score=scores.health_score,
sleep_score=scores.sleep_score,
# etc.
```

**Step D — Update `dialogue.py` / assessment endpoint**

Any reference to `wearable.respiratory_rate_bpm` in the assessment prompt must be updated to `devices.bio_monitor.breathing_rate_bpm`. Check all string formatting in `main.py` and `dialogue.py`.
