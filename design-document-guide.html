Writing Guide

System-Level Design Concept Document

How professional PMs and systems engineers structure this document for HMI/HSI-focused software

Scope of this guide: The document covers initial system-level design concept writing focused on human interface design and human-system interaction (HSI), bridging the gap between a ConOps/requirements baseline and detailed design.

What Is This Document?

The System-Level Design Concept (sometimes called the System Design Description, SDD, or Preliminary Design Document) is the first design artifact that translates operational and human factors requirements into a coherent architectural approach. It answers the question: "How will this system be built at a conceptual level?"


It sits between the ConOps (what the system does operationally) and the detailed design specification (how each component is precisely built). Its primary audiences are:

  • Systems engineers — to verify requirements traceability
  • HMI/UX designers — to establish interaction paradigms
  • Stakeholders and reviewers — to approve the design direction before expensive implementation begins
  • Future maintainers — to understand design intent and key trade-offs

The "Top-Down, Outside-In" Rule

Professional document authors follow a strict ordering logic: always move from context → structure → behavior → appearance → rationale. This mirrors how readers build mental models.

→
Architecture
What pieces
→
Behavior

A Critical Distinction: Concept vs. Specification

This document describes concepts and intentions, not precise specifications. Avoid pixel-perfect measurements, specific algorithm parameters, or implementation-level detail. The goal is to communicate design direction with enough fidelity to allow stakeholder review, not to serve as a build specification.

✅ Include in This Document

  • Functional block diagrams
  • Conceptual wireframes / sketches
  • System context diagrams
  • User task flow diagrams
  • Design decision tables
  • Interface architecture maps
  • Subsystem relationship tables

❌ Defer to Later Documents

  • Pixel-level UI specifications
  • Algorithm or database schemas
  • Test procedures
  • Detailed API definitions
  • Final color/font specifications
  • Implementation timelines
  • Code architecture patterns
Overview

Governing Principles

The mental models professional PMs use when organizing this document

🔭

1. Zoom-In Progressively

Each section narrows the scope. Start at mission/enterprise level, then system level, then subsystem level, then interface level. Never start by showing a detailed screen before showing where it fits.

🗺️

2. Orient Before You Describe

Every major section should open with a diagram or table that orients the reader spatially or structurally before narrative text explains the details.

👤

3. Human First in HSI Documents

In an HSI-focused document, always describe the human role, tasks, and goals before describing the system response. The system architecture exists to serve the operator.

🔗

4. Requirements Traceability Throughout

Every design choice should reference a requirement or ConOps statement. Even at conceptual level, the reader must see the link between a design decision and its functional or human-factors driver.

⚖️

5. Alternatives Are Not Failures

Professional documents explicitly present alternatives that were not selected and explain why. This demonstrates rigor, builds reviewer trust, and protects design decisions in review.

📐

6. One Diagram, One Message

Each diagram should communicate exactly one concept. Do not combine architecture, data flow, and interaction in one figure. Readers should be able to extract the key point in under 10 seconds.

🎯

7. Align Diagram Complexity to Audience

Executive summary figures use high-level context diagrams. Technical sections use detailed block diagrams. The same system should be drawn at the appropriate abstraction level for its audience.

📎

8. Tables Complement Diagrams

Every significant diagram should be accompanied by a table that names, defines, and cross-references its elements. Diagrams convey structure; tables convey completeness and traceability.

NASA/Aerospace-specific note: In systems designed under NASA-STD-3001 or similar human systems integration standards, the document must additionally show how design choices address specific human performance parameters — workload, situational awareness, error tolerance — not just functional requirements.

Section 1

Introduction & Scope

Framing the document for its audience

Purpose of This Section

This section is purely administrative and framing. It should be concise. Its goal is to tell the reader exactly what the document is, what it covers, and what it does not cover — before they invest time reading it.

What to Include

  • Document purpose: One paragraph stating why this document exists and what decision it supports.
  • Scope statement: What system functions and design aspects are covered. Explicitly state what is out of scope.
  • Relationship to other documents: A short lineage table (ConOps → Requirements → this document → future SDD).
  • Intended audience: Who this document is written for and what they are expected to do with it.
  • Assumptions and constraints: Any assumptions made about the operational environment, hardware platform, or user population.
  • Glossary and acronyms: Or a reference to the master acronym list.

Typical Visual: Document Lineage Table

A simple table showing parent/sibling documents and their relationship to this one.

DocumentTypeRelationship
ConOpsParentDefines operational intent that this design must fulfill
Requirements BaselineParentDefines mandatory constraints this design must satisfy
This Document (Design Concept)CurrentTranslates requirements into conceptual architecture
Detailed Design SpecChildWill refine concepts herein into build specifications
Common mistake: PMs often over-write Section 1, turning it into a second ConOps. Keep this section under one page. If the reader needs operational background, cite the ConOps — do not repeat it.
Section 2

Operational Context & User Overview

Establishing the system boundary and who operates it

Why this comes second: Before describing any architecture, the reader must understand the system's external environment — what it connects to, who uses it, and under what conditions. This is the "outside" before the "inside."

Context Diagram (Level 0) — First Diagram in the Document

The context diagram is the most important figure in the entire document. It establishes the system boundary: what is inside vs. outside, and what external entities the system exchanges information with. Draw it first; write the rest of the document to explain it.


For an HMI-focused document, the context diagram should show:

  • The system (as a single box)
  • Human users (by role) as external actors
  • Other systems that provide data inputs (e.g., wearable sensors, environmental systems)
  • Other systems that receive outputs (e.g., ground medical teams, mission control)
  • Labeled data/information flows on each arrow

User Role Table

Immediately follow the context diagram with a table that formally defines each user role shown in it.

RolePrimary TasksAccess LevelHuman Factors Considerations
CrewmemberMonitor personal health status; acknowledge alerts; initiate backupsFull operationalHigh workload, constrained environment, limited expert support
Medical Officer (Crew)Interpret trends; generate health reports; configure thresholdsElevated / clinicalTrained medical user; may operate under stress or fatigue
Ground Medical TeamRemote access; analysis; parameter adjustmentRemote / supervisedCommunication latency; no direct environment access

Operational Scenarios Summary

List 3–6 representative operational scenarios that will drive the design. These are not full task analyses — those come in Section 5. They are concise scenario titles that allow the reader to understand the range of use cases.

  • Nominal daily health monitoring (no alerts)
  • Physiological threshold exceedance — alert acknowledgement flow
  • Degraded mode operation (sensor failure, partial data)
  • Ground-supported mode — asynchronous remote access
  • Pre-EVA health clearance check
  • Emergency: sudden critical alert during high crew workload
HSI-specific requirement: For human-centered systems, the user role table must include human factors considerations — not just functional access levels. Cognitive load, training background, environmental constraints, and potential stress conditions belong here.
Section 3

System Architecture Overview

Identifying major subsystems and their relationships

Why this comes third: Only after the reader understands the external context (§2) can they meaningfully understand the internal structure. This section reveals what subsystems make up the system and how information flows between them.

Functional Block Diagram — The Architectural Centerpiece

The functional block diagram (FBD) is the primary architectural visual. It shows all major subsystems as labeled boxes with the information/data flows between them as arrows. Guidelines:

  • Each block = one major subsystem (not a module or component)
  • Arrows carry labeled data/information, not function calls
  • Group related subsystems visually (e.g., data acquisition layer, processing layer, presentation layer)
  • Keep it to 5–10 blocks; more detail belongs in §4
  • Use a consistent left-to-right or top-to-bottom reading direction

Section 4

Subsystem Decomposition & Functions

Drilling down into each major subsystem

Structure: Repeat a consistent sub-structure for each major subsystem. This makes the document scannable and ensures completeness. A reader should be able to look up any subsystem and find the same categories of information.

Recommended Per-Subsystem Sub-Structure

For each subsystem identified in §3, provide the following in a consistent format:

  1. Purpose and scope — 2–3 sentence description of what this subsystem does and why it exists.
  2. Key functions — Bulleted list of major functions (not implementation methods).
  3. Inputs and outputs — Table of data flows into and out of the subsystem.
  4. Interfaces to other subsystems — Which subsystems does this one depend on or serve?
  5. Design constraints — Any constraints from requirements or standards that apply specifically to this subsystem.

Interface Matrix (N×N)

After all subsystems are described, include a single interface matrix showing all inter-subsystem communication. This is a grid where rows and columns are subsystems, and cells indicate the type of information exchanged.


This is one of the most referenced tables in the entire document. Reviewers use it to verify that no required data flows have been missed and that the architecture is internally consistent.

Function Allocation Table

For HSI-focused documents, include a function allocation table that explicitly assigns each major system function to either the system (automated) or the human operator (manual), or a shared allocation. This is required under most human systems integration frameworks.

FunctionAllocationRationaleReq. Reference
Continuous data collectionSystem (auto)Continuous, background; no crew action neededREQ-001
Threshold exceedance detectionSystem (auto)Faster than human response; safety criticalREQ-006
Alert acknowledgementHuman (manual)Human judgment required for contextual validityREQ-007
Health trend interpretationSharedSystem provides scores; human makes final decisionREQ-005
Data backup initiationSharedAuto backup continuous; crew can initiate on demandREQ-010
Common mistake: Teams often omit the function allocation table from design documents, treating it as a requirements artifact. In HSI design, it belongs here because the allocation directly drives interface complexity — anything allocated to the human requires an HMI element.
Section 5

Human–System Interaction Design

The behavioral heart of an HSI-focused document

Why this is the most important section for HSI: Sections 2–4 describe what the system is. This section describes how humans and the system exchange information and control. This is where the human factors requirements (workload, situational awareness, error prevention) are first addressed at a design level.

5.1 Interaction Paradigm Statement

Before any diagrams, write a clear prose statement of the overall interaction paradigm. This is the design philosophy that unifies all interaction decisions. Examples:

  • "Monitoring-first, action-on-demand" — the system continuously presents status passively; the crew initiates actions only when needed.
  • "Alert-driven with progressive disclosure" — the primary interaction is alert acknowledgement; detailed data is one navigation step away.
  • "Tiered information density" — nominal state shows minimal data; anomalies progressively surface additional context.

This statement should be a single paragraph that a reviewer can refer back to when evaluating every subsequent design decision.

5.2 User Task Flow Diagrams

For each representative scenario identified in §2, create a user task flow diagram. Format: swimlane diagram (one lane per actor: Crew / System / Ground Team). Show:

  • Human actions as rectangles
  • System responses as rounded boxes
  • Decision points as diamonds
  • Information displayed to the user on arrows between system and human lanes
  • Time direction top-to-bottom
Key principle: Task flow diagrams in HSI documents should show information exchange, not just action sequence. Every arrow from system to human lane should be labeled with what information the human receives and in what format.

5.3 Information Architecture Diagram

Sometimes called a "screen hierarchy" or "display map." This diagram shows all major views/screens of the system in a hierarchical tree, with labeled navigation paths between them. It answers the question: "How does a user get from anywhere to anywhere in the system?"


For an alert-driven system, also show alert-triggered navigation paths (e.g., "from any screen → alert modal → detail view") separately from normal navigation paths.

5.4 Mode Transition Diagram

A state machine diagram showing all operational modes, the events that trigger transitions between them, and the system and HMI behaviors in each mode. Required because requirements often mandate explicit mode display (as in this project's MODE VISIBILITY requirement).


Modes to consider for a health monitoring system:

  • Nominal Monitoring Mode — all systems functional, no active alerts
  • Alert Mode — one or more active threshold exceedances requiring crew attention
  • Degraded Mode — one or more sensors unavailable; system operating on partial data
  • Ground-Supported Mode — remote access active; crew notified of ground operations

5.5 Alert & Notification Hierarchy

A dedicated diagram showing the alert taxonomy — priority levels, associated display behaviors (color, sound, position), and required crew actions per level. This directly supports requirements for alert prioritization and design-induced error prevention.


Present as both a diagram (visual hierarchy) and a table (formal specification of behaviors per level).

HSI linkage: Every task flow diagram should end with a notation of which human performance requirements it is intended to satisfy (workload, situational awareness, error tolerance). This links behavioral design to requirements before the interface is even drawn.
Section 6

Interface Design Concepts

First visual representation of screens and layouts

Why wireframes come this late: Screens and wireframes are the output of the design decisions made in sections 2–5. Presenting them earlier — before context, architecture, and interaction behavior are established — turns them into mere visual ideas rather than justified design outcomes. A reviewer evaluating a wireframe in §6 should already know why it exists, from the preceding sections.

6.1 Display Hierarchy & Zone Layout

Before presenting specific screen wireframes, show the conceptual display zone model — a labeled sketch of how the screen is divided into functional zones. Common zones for monitoring systems:

  • Status header zone — current mode, system health, time/mission elapsed
  • Primary data zone — the main content area for the current view
  • Alert/notification zone — persistent area for active alerts regardless of current view
  • Navigation zone — access to major functional areas
  • Contextual action zone — view-specific controls and actions

This zone model should appear in every subsequent wireframe, confirming consistency across views.

6.2 Key View Wireframes / Conceptual Sketches

Select 4–6 key views that cover the most critical or representative interaction scenarios. Do not wireframe every possible screen — only the ones that establish the design language. Required coverage:

  • Home / Dashboard view — nominal state overview
  • Alert detail view — the most critical interaction flow
  • Historical trend view — a data-dense view establishing density conventions
  • Mode indicator / status view — demonstrating mode visibility design
  • Settings / configuration view (optional) — if crew-modifiable parameters are significant
Fidelity rule: Wireframes in this document should be mid-fidelity — clear enough to communicate layout and information hierarchy, but not so detailed that reviewers debate color choices or exact button labels. Use grayscale, labeled boxes, and representative placeholder content.

6.3 Navigation Flow Diagram

Complement the information architecture diagram from §5 with a navigation flow that shows the path a user takes through the most critical task flows. Unlike the information architecture (which shows all screens), this diagram shows specific paths for specific scenarios — highlighting which views are most central and most frequently accessed.

6.4 Visual Design Language Intent

A brief summary (not a full style guide) of the visual design intentions. This establishes design language at a level sufficient for concept review:

  • Color usage intent (e.g., semantic color coding for alert levels per NASA-STD-3001)
  • Typography intent (e.g., high-legibility typeface for varied lighting conditions)
  • Iconography approach (e.g., standard labeling plan per requirements)
  • Information density philosophy (e.g., progressive disclosure principles)
Link back to requirements: Each wireframe or design language element should cite the requirement it satisfies (e.g., "Alert color coding → REQ-007 ALERT PRIORITIZATION"). This traceability transforms wireframes from design artifacts into requirement evidence.
Section 7

Key Design Decisions & Rationale

Documenting trade-offs and justifying the selected approach

Why this comes last: Design decisions can only be meaningfully evaluated after the reader understands the full architecture and interaction design. Placing rationale last also makes it easy to reference specific elements from §3–§6 when explaining why they were chosen.

Design Decision Matrix — The Key Artifact of This Section

The design decision matrix is a structured table capturing every significant design choice. It is the primary document of record for stakeholder review. A well-formed matrix prevents the document from becoming a justification narrative — it forces precision.

IDDecision TopicOption SelectedAlternatives ConsideredRationaleReq. Driver
DD-01 Alert display paradigm Persistent alert bar (always visible) Modal overlay; notification drawer; ambient indication Persistent visibility required for situational awareness; modal overlay introduces display interruption risk during high workload tasks REQ-005, REQ-007
DD-02 Mode indication approach Full-screen header band color change + text label Icon only; color only; separate status display panel Color alone is insufficient per legibility requirements; combined text + color redundancy reduces error under adverse lighting REQ-004, REQ-006
DD-03 Data presentation density Progressive disclosure — summary by default, detail on demand Maximum data display; tabbed detailed views; user-configurable layouts Reduces nominal cognitive workload (≤ Bedford 3); preserves access to full data when needed; user-configurable layouts introduce training burden REQ-003, REQ-005

What Counts as a "Significant" Decision?

Include a decision in the matrix if it meets any of these criteria:

  • It directly affects human performance (workload, SA, error risk)
  • Multiple viable alternatives existed
  • It is driven by a specific human factors or safety requirement
  • A reviewer might reasonably question why another option was not chosen
  • It will constrain future design decisions

Trade Study Summaries (When Applicable)

For decisions where quantitative comparison was performed (e.g., usability evaluation of competing navigation patterns, weighted criteria matrices), provide a brief summary of the trade study method and results. The full analysis may be in an appendix.

Requirements Traceability Summary

Close the document with a summary traceability table showing how each top-level requirement is addressed by at least one design element described in the document. This is not a full requirements traceability matrix — that belongs in the formal SDD. It is a sanity check that confirms every requirement has been considered at the design concept level.

Common mistake: Teams write rationale as "we chose Option A because it is better." Professional rationale must state: better with respect to what criterion, under what constraints, for which requirement. Rationale without criteria is opinion.

Which diagram type to use, when, and why

The rule of visual selection: Choose the diagram type that most directly answers the primary question a reader would ask at that point in the document. Different diagram types answer different questions.

The Diagram Companion Rule

Every diagram must be accompanied by either a table or a prose description that names and defines its elements. A diagram without text is ambiguous; text without a diagram is hard to visualize. The combination is always stronger than either alone.

When NOT to Use a Diagram

  • When the information is already captured in a table and a diagram would add no spatial or relational clarity
  • When there are fewer than 3 elements to show (a list is clearer)
  • When the diagram would require so many annotations it becomes harder to read than prose
  • In §1 (Introduction) — no diagrams; this section is administrative

HMI / HSI-Specific Advice

What makes a design document HSI-focused, not just technically complete

The HSI difference: A technically complete design document describes what the system does. An HSI-complete document additionally describes what the human must do, perceive, and decide — and how the design supports that. The human is a subsystem and must be treated as one.

1. Always Start with the Human, Not the System

When writing any section that involves human-system interaction, write the human's task and goal first, then describe the system's role in supporting it. This forces the author to maintain a user-centered perspective and prevents the document from becoming technology-first.

2. Make Human Performance Requirements Visible in Design Sections

Requirements for workload (Bedford ≤ 3 nominal), situational awareness (SAGAT/SART), and usability (NMSUS ≥ 85) should be cited at the point in §5 and §6 where the design addresses them — not only in the requirements document. This makes the design intent legible to evaluators.

3. Document the "Information Needs" Explicitly

For each major user task, identify: What information does the user need? When do they need it? In what format? This is sometimes called an "information requirements analysis" and is distinct from functional requirements. It directly drives display design.

User TaskInformation NeededWhen NeededRequired FormatSource Subsystem
Acknowledge health alertAlert type, severity, affected parameter, current value, thresholdImmediately on alert triggerPrioritized visual + auditory notification with numerical valuesSS-03
Pre-EVA health clearanceAll current vital signs, trend over last 24h, radiation dose, fatigue scoreOn-demand, within 1 min of decisionSummary dashboard with trend sparklines and clearance statusSS-03, SS-04

4. Address Cognitive Load Architecture, Not Just Functional Architecture

Describe specifically how the system architecture reduces cognitive load. Examples of design elements that address cognitive load at the architectural level:

  • Progressive disclosure: Summary data by default; detail requires explicit navigation
  • Consistent spatial mapping: Alert information always in the same screen zone, regardless of current view
  • Predictive indicators: Risk scores and readiness indices replace raw data interpretation burden
  • Automated anomaly detection: The system identifies threshold exceedances; the human only evaluates flagged events

5. Show Error Pathways, Not Just Happy Paths

Include at least one task flow diagram showing what happens when things go wrong: a false positive alert, a degraded sensor condition, or an interrupted task. This demonstrates that the design has considered error resilience, which directly addresses design-induced error requirements.

6. Explicitly Address Legibility Across Operational Conditions

For systems operating in variable environments (lighting, vibration, viewing angle), the design concept document should acknowledge legibility constraints at the conceptual level — even before specific font sizes are chosen. A statement such as "all primary status indicators shall use redundant coding (color + shape/text)" establishes a design principle that constrains all subsequent interface decisions.

NASA-STD-3001 alignment: When writing for a NASA human systems context, map your document sections to specific Volume 2 requirements areas. Reviewers will evaluate the document against the standard. If your §5 (HSI) section does not visibly address §V2 5006 (Situational Awareness), §V2 5007 (Workload), and §V2 10001 (Usability), expect review comments regardless of how technically complete the rest of the document is.

Document Completion Checklist

Use this before submitting for stakeholder review

§1 Introduction

  • Document purpose stated in ≤ 1 paragraph
  • Scope and out-of-scope explicitly stated
  • Relationship to ConOps and requirements documented
  • Intended audience identified
  • Key assumptions and constraints listed

§2 Operational Context

  • Context diagram (Level 0) present and labeled
  • All external actors identified with data flows
  • User role table with human factors column
  • Representative operational scenarios listed

§3 Architecture Overview

  • Architecture layering or approach described in prose first
  • Functional block diagram present
  • All subsystems labeled and identified in the FBD
  • Subsystem identification table accompanies FBD
  • Data flows labeled on diagram arrows

§4 Subsystem Decomposition

  • Consistent sub-structure used for each subsystem
  • Inputs/outputs defined per subsystem
  • Interface matrix included
  • Function allocation table (human vs. system) included

§5 Human–System Interaction

  • Interaction paradigm statement written
  • Task flow diagrams for key scenarios (including error case)
  • Information architecture diagram present
  • Mode transition diagram covers all required modes
  • Alert hierarchy diagram and table present
  • Information needs analysis table included
  • Human performance requirements cited in relevant subsections

§6 Interface Design Concepts

  • Display zone model established before wireframes
  • Key view wireframes present (4–6 minimum)
  • Wireframes use consistent zone model
  • Navigation flow diagram shows critical task paths
  • Visual design language intent documented
  • Each wireframe cites the requirement it addresses

§7 Design Decisions

  • Design decision matrix present with alternatives column
  • All rationale entries cite a requirement or criterion
  • Trade studies summarized where applicable
  • Requirements traceability summary present

Overall Document

  • Every diagram has a companion table or labeled prose
  • Top-down ordering maintained throughout
  • No implementation-level specifications included
  • Human appears as an actor in every interaction diagram
  • NASA-STD-3001 Vol. 2 requirements visibly addressed
